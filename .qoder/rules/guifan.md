# Diary App 项目规范（Flutter）

## 1. 目标与原则
- 本项目的第一目标是可维护，其次是稳定性，最后才是开发速度。
- 所有改动必须优先考虑长期维护成本，禁止用临时堆叠代码完成需求。
- 任何功能优化不得牺牲用户核心体验（例如不能因性能问题直接删除核心功能）。

## 2. 目录与职责
- `lib/screens`：页面壳、路由和页面级编排，不承载复杂业务逻辑。
- `lib/widgets`：可复用 UI 组件、页面局部组件。
- `lib/services`：数据持久化、AI 调用、外部接口、系统服务封装。
- `lib/models`：数据模型和序列化逻辑。
- `lib/theme`：主题、配色、亮暗模式与全局视觉 token 定义。
- `lib/utils`（如有）：纯工具函数，不依赖页面状态。

## 3. 文件规模约束
- 单个 Dart 文件建议不超过 500 行。
- 单个 Dart 文件超过 700 行时，新增功能前必须先做拆分。
- 单个 `build()` 方法建议不超过 120 行，超出时必须提取子组件。
- 新功能优先新增文件，不继续向超大文件堆逻辑。

## 4. 页面与状态规范
- 页面层只处理展示与交互触发，不写复杂数据处理。
- 业务逻辑放入 service 或 controller（如后续引入）。
- 避免在 `build()` 中进行重计算、重复遍历、重复 JSON 解析。
- 异步操作必须包含：
  - 错误处理（`try/catch`）
  - `mounted` 判断（涉及 `setState` 或 UI 反馈时）
  - 用户可理解的失败提示

## 5. UI 一致性规范
- 优先复用统一样式 token（如 `AppUi`），禁止到处写硬编码间距/圆角/颜色。
- 间距遵循统一节奏（如 8/12/16），卡片圆角统一。
- 提示样式统一使用顶部提示组件（如 `AppTopToast`），避免多种提示并存。
- 深色模式与浅色模式都必须验证可读性与对比度。
- 新增 UI 必须与现有设计语言一致，不引入突兀配色。

## 6. 性能规范（重点）
- 列表默认使用 `ListView.builder` / 懒加载，避免 `Column + map + toList` 渲染大集合。
- 图片场景必须控制解码尺寸（`cacheWidth/cacheHeight`），避免原图直接参与高频渲染。
- 滚动场景避免在每帧触发高成本逻辑，优先使用缓存和预计算。
- 日历、抽屉、聊天流等高频区域必须优先做局部重建，降低整页 `setState` 影响面。
- 性能优化以“保留功能”为前提，采用渐进加载、优先级加载和缓存策略。

## 7. AI 开发流程规范
- 每次编码前必须先输出：
  - 改动计划
  - 涉及文件清单
  - 风险点和回滚点
- 每次编码后必须输出：
  - 实际改动文件
  - 核心修改说明
  - 用户可感知变化
  - 验证结果（诊断/分析）
- 若目标文件超大（>700 行），优先执行“最小拆分”后再加功能。

## 8. 质量与验证
- 完成功能后至少执行一次与改动相关的静态检查（如 `dart analyze` 指定文件）。
- 发现历史 warning 需标注“历史遗留”或“本次引入”，不得混淆。
- 关键修复（数据一致性、保存、删除、恢复、AI 调用）必须做最小路径回归验证。

## 9. Git 与提交规范
- 重要改动前先创建 checkpoint 提交，便于回滚。
- 提交信息格式建议：
  - `feat: ...`
  - `fix: ...`
  - `refactor: ...`
  - `perf: ...`
  - `chore: ...`
- 禁止在一次提交中混入大量无关改动。

## 10. 禁止事项
- 禁止把多个职责长期堆在同一个页面文件中。
- 禁止为了“看起来完成了”而忽略性能和结构问题。
- 禁止未经说明的大范围重构。
- 禁止删除用户可见功能来规避技术问题。
- 禁止新增与当前任务无关的依赖或架构变更。
